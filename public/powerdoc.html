PowerDoc for meagent

核心理念 数据结构 构建阶段 工具链 审查推演

超大型 .docx 合同的智能解构与审查

摒弃传统切块与相似度盲搜。通过构建 Agentic Tree-Graph (树图混合索引)，赋予 meagent 像人类高级法务一样的“导航、推理与精读”能力。

核心理念：面向文档的结构化索引 (Tree-Graph Index)

传统 RAG 把 30 万字切成无数碎片，通过向量相似度强行匹配，极易导致“上下文断裂”。

PowerDoc 采用的是一种 面向文档的结构化索引方案（类似 PageIndex）。它保留长文档的原生树状结构（目录、章节），并提取各节点摘要。Agent 遇到问题时，先看目录树，推理出应该去哪个章节找答案，然后直接提取目标章节的原文。在此基础上，我们进一步引入了图关联（Graph）解决条款交叉引用，并辅以向量检索作为兜底排雷策略。

❌ 传统大模型 / 向量 RAG 痛点

  • ✖ 上下文打爆： 30W字直接灌入，推理慢、极易遗漏关键条款。
  • ✖ 逻辑割裂： 向量 Chunking 让模型不知道“这句话属于主合同还是附件”。
  • ✖ 无法处理交叉引用： 遇到“依据第3.2条规定”，向量检索彻底失效。
PowerDoc 方案

✅ PowerDoc (增强型结构化索引)

  • ✔ 原生目录导航： Agent 第一步看结构树，推理定位目标章节。
  • ✔ Tree-Graph 融合： 独创节点边关系，解决条款交叉引用。
  • ✔ 按章节无损读取： 直接调取节点原文，保留完整段落逻辑。
  • ✔ 向量库双保险： 逐行向量库仅作为“防漏检”兜底工具。

核心数据结构：带有引用边的树形索引 (Tree-Graph Index)

这是 PowerDoc 的核心大脑。它不仅保留了 docx 原生标题的层级（Tree），还在节点上存储了该条款所引用的其他节点关联（Graph Edge）。

构建阶段：Sub-Agent 异步编排流

利用 docx 原生的 Heading XML 结构。后台通过低成本的 Sub-agents 预先构建好这套结构化索引。

PowerDoc 审查工具链 (暴露给 meagent)

get_toc(max_depth) #树检索起点

功能：获取合同骨架视图

Agent 第一步。返回文档的树形结构及节点 ID。这让 Agent 宏观把握合同结构，决定第一步该去哪一个大章节找答案，符合人类法务的阅读习惯。

get_node_info(node_ids) #摘要与关联

功能：嗅探节点摘要与图谱交叉关联

Agent 无需读全文，只看目标节点的 Summary、**行号范围**，以及 **关联节点(Links)**。若发现它与其他条款有依赖，可顺藤摸瓜。

read_nodes(node_ids) #结构化主阅读

功能：提取指定章节的完整原文

Agent 最核心的阅读工具。 当确信某章节（如“第8.2节”）至关重要时，直接传入 Node ID，获取该章节及其子节点的无损原文文本，逻辑连贯。

read_lines(start, end) #细粒度精读

功能：通过行号范围抽取片段

补充阅读工具。如果一个节点包含上千行内容，Agent 可以通过 get_node_info 获取到的局部行号范围，仅拉取所需的十几行文本，极致节省上下文。

semantic_search(query, filter_node) #向量兜底排雷

功能：防漏检全局/局部语义搜索

结构化推理解析完成后，为防止恶意条款隐藏在“杂项”等不相关的树节点中，Agent 使用自然语言进行全局或指定章节内的模糊向量搜索，作为整个审查流程的安全兜底。

实战推演：基于结构化索引的“数据泄露责任”排查

场景：主合同规定违约上限为 100 万，但这是一个复杂的系统合同，我们需要查清底层“是否存在对乙方不利的数据泄露惩罚例外”。
